# Product Context - MTS MultAgent Employee Monitoring System

## 🎯 Почему этот продукт существует

MTS MultAgent Employee Monitoring System создана для решения критической проблемы управления большими командами в корпоративной среде MTS. Руководители отделов тратят до 20 часов в неделю на ручной анализ производительности сотрудников, что ведет к:

- **Субъективным оценкам** вместо данных-driven решений
- **Запаздыванию в выявлении проблем** на недели и месяцы
- **Выгоранию менеджеров** от рутинных операций
- **Потере производительности** из-за неэффективного управления

## 💡 Какие проблемы решает

### 1. **Автоматизация рутинного анализа**
**Проблема:** Руководители вручную собирают данные из Jira, анализируют протоколы совещаний, считают метрики
**Решение:** Система автоматически собирает и анализирует все данные ежедневно

### 2. **Объективная оценка производительности**
**Проблема:** Личные отношения и субъективное восприятие влияют на оценки
**Решение:** LLM-анализ обеспечивает объективную оценку на основе фактических данных

### 3. **Раннее выявление рисков**
**Проблема:** Проблемы обнаруживаются когда уже поздно исправлять
**Решение:** Ежедневный мониторинг показывает тренды и отклонения в реальном времени

### 4. **Экономия времени руководителей**
**Проблема:** Менеджеры тратят время на сбор данных вместо стратегических задач
**Решение:** Готовые отчеты экономят 15-20 часов в неделю на каждого руководителя

## 🎬 Как продукт должен работать

### **Пользовательский сценарий (руководитель отдела):**

1. **Утро 9:00** - Система автоматически собирает данные о задачах команды из Jira за предыдущий день
2. **Утро 10:00** - Руководитель получает краткий отчет:
   - Кто из команды завершил задачи
   - Кто имеет просрочки
   - Текущая загрузка команды
3. **В течение дня** - Система анализирует протоколы совещаний и обновляет данные об активности
4. **Пятница 18:00** - Автоматически публикуется еженедельный отчет в Confluence с:
   - Комплексной оценкой каждого сотрудника
   - Рекомендациями по развитию команды
   - Выявлением success-кейсов и проблемных зон

### **Технический сценарий (автоматическая работа):**

1. **TaskAnalyzerAgent** запускается по расписанию, подключается к Jira API
2. **MeetingAnalyzerAgent** сканирует папки с протоколами, обрабатывает новые файлы
3. **QualityValidatorAgent** проверяет полноту и корректность данных
4. **WeeklyReportsAgent** accumulates данные за неделю и публикует анализ
5. **Orchestrator** координирует все процессы, обрабатывает ошибки

## 🎯 Цели пользовательского опыта

### **Для руководителей:**
- **Простота:** Отчеты должны быть понятны без технических знаний
- **Своевременность:** Данные доступны когда нужно для принятия решений
- **Действенность:** Отчеты содержат конкретные рекомендации, а не только данные
- **Надежность:** Система работает без сбоев и вмешательства

### **Для технической команды:**
- **Масштабируемость:** Легко добавлять новые отделы и сотрудников
- **Интегрируемость:** Простое подключение к другим системам MTS
- **Поддерживаемость:** Чистый код и понятная архитектура
- **Мониторинг:** Прозрачная работа всех компонентов системы

## 🎨 Психология пользователя

### **Барьеры adoption:**
- **Недоверие к автоматизации** - боязнь что LLM неправильно оценит людей
- **Сопротивление изменениям** - привычка к ручным процессам
- **Страх потери контроля** - опасения что система сделает решения за руководителя

### **Как мы их решаем:**
- **Прозрачность:** Все метрики и выводы объясняются
- **Human-in-the-loop:** Руководитель всегда может скорректировать оценку
- **Постепенное внедрение:** Система дополняет, а не заменяет интуицию
- **Обучение:** Документация и поддержка для освоения

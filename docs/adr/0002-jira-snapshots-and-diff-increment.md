# ADR-0002: Jira Snapshots и Diff как инкремент изменений\n\n## Статус\nПринято\n\n## Контекст\nСистема генерирует анализ задач Jira и протоколов встреч и сохраняет артефакты в `reports/runs/{run_id}` и `reports/latest`.\n\nДля сравнительного анализа (\"новый анализ\" vs \"старый анализ\") и построения динамики по сотрудникам требуется:\n- фиксировать исходные данные Jira на момент запуска (snapshot)\n- иметь машинно-обрабатываемую дельту между запусками (diff)\n- хранить это как часть run-артефактов, чтобы последующие агенты могли использовать инкремент.\n\n## Решение\n1) Вводится сущность \"Jira snapshot\": JSON файл с метаданными и нормализованным списком задач.\n- Путь хранения данных: `data/jira/snapshots/{run_id}.json`\n\n2) Вводится сущность \"Jira diff\": JSON файл с результатом сравнения текущего snapshot с предыдущим.\n- Путь артефакта run: `reports/runs/{run_id}/jira-diff/jira-diff.json`\n- Путь артефакта latest: `reports/latest/jira-diff/jira-diff.json`\n\n3) Ответственность за формирование snapshot/diff принадлежит `ImprovedTaskAnalyzerAgent`.\n\n## Последствия\nПлюсы:\n- появляется строго определённый инкремент (added/removed/changed) между запусками\n- упрощается построение динамики команды: изменения задач можно агрегировать по assignee/status\n- артефакты становятся воспроизводимыми (snapshot фиксирует входные данные)\n\nМинусы/ограничения:\n- diff не является глубоким доменным (пока используется минимальный fingerprint полей)\n- при изменении схемы нормализации задач потребуется миграция/версионирование snapshot\n\n## Альтернативы\n- сравнивать только итоговые отчёты LLM: отвергнуто, т.к. LLM вывод не гарантирует стабильность и полноту\n- хранить снапшоты внутри `reports/runs`: отвергнуто, т.к. снапшоты являются данными-источниками и логичнее держать их в `data/`\n
